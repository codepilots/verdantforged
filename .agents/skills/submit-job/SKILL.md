# TEE Broker Job Submission

This skill provides step-by-step instructions and reference commands for submitting computation or LLM-proxy jobs to the VerdantForged TEE Broker.

## When to use this skill

Use this skill when:
- You need to run a task/job (e.g. `summarize`, `code-review`, `photo-glow-up`) on the attested worker enclaves.
- You are testing the client-to-broker integration on a local/remote environment.
- You need to perform a test transaction through the demo payment gateway.

Do NOT use this skill when:
- You are setting up or configuring the broker daemon infra (use `tee-broker-deploy-config`).
- You are checking whether the broker's enclave attestation is cryptographically valid (use `verify-attestation`).

## 1. Demo Mode (Stub Payment Path)

In demo mode, you can bypass the Stripe payment gateway by minting a mock Shared Payment Token (SPT). This is suitable for general API testing, development, and system health validation.

### Step 1: Mint a mock SPT
Request a demo token from the broker's payment stub route:
```bash
SPT=$(curl -sS -X POST https://verdant.codepilots.co.uk/v1/demo/shared-payment-token \
  -H 'Content-Type: application/json' -d '{}' | jq -r .shared_payment_token)
```

### Step 2: Submit the job payload
Send the payload to `POST /v1/jobs` with the mock payment token:
```bash
curl -sS -X POST https://verdant.codepilots.co.uk/v1/jobs \
  -H 'Content-Type: application/json' \
  -d "{
    \"client_req_id\":   \"demo-job-$(date +%s)\",
    \"encrypted_skill\": \"summarize\",
    \"encrypted_data\":  \"VerdantForged is a secure, low-cost TEE broker control plane architecture.\",
    \"requester_sig\":   \"0x\",
    \"result_pubkey\":   \"0x\",
    \"shared_payment_token\": \"$SPT\"
  }"
```
This returns a job response containing the `job_id` and the `job_access_token` used for subsequent queries.

## 2. Production Mode (Encrypted Path)

In production, inputs are encrypted using **X25519** and signed with the client's key. The broker creates a real Stripe PaymentIntent, requires client payment, and queues the job only once payment is verified.

### Payload Structure
Submit the request to `POST /v1/jobs` with:
- `client_req_id`: Unique identifier generated by the client.
- `encrypted_skill`: Symmetric or asymmetric encrypted name of the skill to execute.
- `encrypted_data`: Encrypted task data/payload.
- `requester_sig`: Hex-encoded cryptographic signature proving request ownership.
- `result_pubkey`: Asymmetric public key used by the worker to encrypt output artifacts.
- `stripe_pi_id` or `shared_payment_token`: Verified payment intent ID or Stripe Connect token.

### Checking Job Status & Artifacts
To monitor progress or retrieve final outputs, authenticate using the job's access token:
```bash
JOB_ID="job_xxx"
JOB_TOKEN="jobtok_xxx"

# Retrieve job status (e.g. queued, running, completed)
curl -H "Authorization: Bearer $JOB_TOKEN" \
  "https://verdant.codepilots.co.uk/v1/jobs/$JOB_ID"

# Fetch artifacts manifest once complete
curl -H "Authorization: Bearer $JOB_TOKEN" \
  "https://verdant.codepilots.co.uk/v1/jobs/$JOB_ID/artifacts"
```
